# Test Strategy

What gets tested at what level, and the target coverage.

---

## Levels

### Unit
- Scope: single function or class in isolation
- Speed: milliseconds
- Mocks: all I/O
- Location: `__tests__/` directories alongside code; Python `tests/unit/`
- Run: `make test-unit` (or `pnpm test:unit` / `pytest tests/unit`)

### Integration
- Scope: service + its direct dependencies (DB, Redis, vault)
- Speed: seconds per test
- Fixtures: docker-compose-backed Postgres + Redis (per-test-run, reset via transactions or truncate)
- Location: `tests/integration/` (Python) and `test/integration/` (TS)
- Run: `make test-integration`

### Contract
- Scope: control-plane ↔ cognition RPC; frontend ↔ control-plane HTTP/WS
- Tool: Zod schemas shared in `packages/schemas/`; verified by generated test fixtures
- Location: `packages/schemas/tests/`
- Run: `make test-contract`

### End-to-End
- Scope: full stack from frontend through control-plane, cognition, executor, vault
- Tool: Playwright for UI flows; k6 for load-ish smoke
- Fixtures: full docker-compose stack (dev profile)
- Location: `tests/e2e/`
- Run: `make test-e2e`

### Safety / Adversarial
- Scope: prompt injection suite, policy engine property tests, blast-radius classifier boundary cases
- Tool: pytest + curated attack corpus (Phase 5)
- Run: `make test-safety`

## Coverage Targets

| Component | Target |
|---|---|
| Policy engine | 100% branch |
| Approval state machine | 100% branch |
| Blast radius classifier | 100% branch |
| Credential vault (server) | 100% branch |
| Capability token service | 100% branch |
| WritePolicyService | 100% branch |
| Safety sanitizer | 100% branch |
| Ego runtime | 85% |
| Task dispatcher | 85% |
| Provider adapters | 80% |
| Memory service | 85% |
| Control plane (overall) | 80% |
| Cognition (overall) | 80% |
| Frontend | 70% (plus critical E2E) |

CI fails on regressions >2% on any module.

---

## Per-Feature Requirements

New feature merges require:
- Unit tests for new code
- Integration test if the feature touches I/O boundaries
- E2E test if the feature is user-observable
- ADR update if architectural
- Doc update if API surface changes

Bug fixes require:
- Regression test that fails on the old code and passes on the new

Safety-critical changes (Layer 0) require:
- 100% branch coverage maintained
- Negative tests for every new policy rule
- Property-based tests where feasible

---

## Test Data

- No production data in tests
- Fixtures generated by factories (`tests/factories/`)
- No static SQL dumps except for explicit migration tests
- Anonymized sample messages for prompt evaluation

---

## Flaky Tests

- Any flaky test is a P2 bug until fixed
- Quarantine via tagging (`@flaky`) with a linked issue
- Quarantined tests don't block CI but are tracked; zero tolerance for backlog growth

## CI Pipeline

### `.github/workflows/ci.yml` (on every PR, fast)
1. Install deps (cached)
2. Lint + typecheck (parallel)
3. Unit tests (parallel)
4. Build Docker images (cached layers)
5. Report status

Target: <5 min.

### `.github/workflows/integration.yml` (on every PR, full)
1. Bring up docker-compose (postgres, redis, vault-dev, minio)
2. Run migrations
3. Integration tests
4. Contract tests
5. Teardown

Target: <15 min.

### `.github/workflows/scan.yml` (nightly + on PRs touching deps)
1. `pnpm audit`
2. `pip-audit` / `uv`-equivalent
3. Trivy scan of built images
4. `detect-secrets` on diff
5. Fail on critical severity

### `.github/workflows/deploy.yml` (on main push)
See [../operations/deploy.md](../operations/deploy.md).

## Local Workflow

```bash
make bootstrap      # one-time
make up             # dev stack
make test           # full suite (unit + integration)
make test-unit      # fast loop
make test-watch     # file-watch mode (TS vitest, Python pytest-watch)
```

## Mocking

- External providers (Anthropic, OpenAI, OpenRouter): mock at the HTTP layer using `httpx.MockTransport` (Python) and `msw` (TS)
- Never mock the policy engine — always run the real one (it's pure code)
- Never mock the vault in integration tests — use the real container

## Property Testing

- TypeScript: `fast-check` for classifier, parser edge cases
- Python: `hypothesis` for self-state MD↔JSON roundtrip, RRF ordering

## Performance

- Per-service latency targets documented in spec + verified in integration suite
- Benchmarks: `pnpm bench` / `pytest tests/benchmarks --benchmark-only`
- CI runs benchmarks weekly, reports regressions

## Prompt Evaluation (Phase 5)

Separate from test suite:
- Curated prompt set per persona
- Run through Ego
- Measure: voice consistency, mode transition correctness, refusal quality
- Reported on a dashboard; not a blocking CI gate

## Chaos Testing (Phase 6)

- Vault restart during tool call
- Postgres restart during run
- Redis loss during approval
- Model provider 503 bursts
- Verify graceful degradation matches documented behavior
